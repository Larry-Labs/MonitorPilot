# MonitorPilot 输入源切换逻辑分析

## 一、DDC/CI 协议能力边界

MonitorPilot 通过 DDC/CI 协议的 VCP Code `0x60`（Input Source）控制显示器输入切换。理解协议能力边界是设计的前提：

| 能力 | 支持情况 | 说明 |
|------|---------|------|
| 读取当前输入源 | ✅ 支持 | 读取 VCP 0x60，返回当前选中的端口值 |
| 设置输入源 | ✅ 支持 | 写入 VCP 0x60，请求切换到目标端口 |
| 检测端口是否有信号 | ❌ 不支持 | DDC/CI 没有标准方法检测某端口是否有活跃视频信号 |
| 检测端口是否物理存在 | ❌ 不支持 | 协议不区分"没有这个端口"和"有端口但没接线" |
| 获取支持的端口列表 | ⚠️ 部分支持 | VCP 0x60 的 maximum 值理论上反映范围，但实际很少准确 |

**关键限制**：DDC/CI 的 `0x60` 读值反映的是显示器**当前选中的输入源**，而非**当前有信号的输入源**。当显示器因无信号自动回退时，DDC 寄存器的更新可能**滞后于视觉变化数秒甚至十几秒**（取决于显示器固件实现）。

## 二、完整场景分析

### 场景 1：目标端口已连接且有信号

**条件**：显示器有该端口 + 端口连接了外部设备 + 设备正在输出视频信号

**典型例子**：Mac 通过 DP-1 连接显示器，PC 通过 HDMI-1 连接显示器。用户从 DP-1 切换到 HDMI-1。

**行为链**：
1. DDC 发送 `set 0x60 = 0x11`（HDMI-1）
2. 显示器接受指令，切换到 HDMI-1
3. HDMI-1 有信号 → 显示器正常显示 HDMI-1 内容
4. DDC 回读 `0x60` → 返回 `0x11`
5. 切换成功

**软件检测**：
- 前端点击后**立即**将按钮更新为目标端口（真乐观更新）
- 后端验证：DDC 回读匹配目标值（全平台两轮：600ms + 1400ms）→ `status: "success"`
- 按钮保持不变（已是正确状态）+ 成功 Toast

**状态**：✅ 完全可靠检测

---

### 场景 2：目标端口存在但无信号（核心难题）

**条件**：显示器有该端口 + 端口没有连接设备，或连接的设备未开机/未输出信号

**典型例子**：HDMI-2 口存在但没插线。用户从 DP-1 切换到 HDMI-2。

**行为链**：
1. DDC 发送 `set 0x60 = 0x12`（HDMI-2）
2. 显示器接受指令，切换到 HDMI-2
3. HDMI-2 无信号 → 显示器短暂黑屏
4. **显示器固件自动回退**到上一个有信号的端口（通常 2-5 秒内完成视觉切换）
5. DDC 回读 `0x60` → **仍然返回 `0x12`**（DDC 寄存器更新滞后）
6. **经过较长时间**（因固件而异，实测 Legion 显示器 > 10 秒）→ DDC 回读才返回实际端口值

**软件检测**：
- 前端点击后**立即**将按钮更新为目标端口（真乐观更新）
- 后端 2 秒验证（全平台两轮：600ms + 1400ms）：DDC 仍显示目标端口 → 返回 `status: "success"` → 按钮保持不变
- 后续轮询（每 3 秒，冷却期后）：**最终**检测到 DDC 更新为原端口 → 静默修正 UI 按钮状态（不弹额外 Toast）

**状态**：⚠️ **当前软件无法在合理时间内可靠区分此场景与场景 1，但通过轮询静默修正保证最终一致性**

**根本原因**：DDC/CI 协议不提供信号检测能力，且 DDC 寄存器更新延迟无法预测。

---

### 场景 3：目标端口不存在（显示器物理上没有该端口）

**条件**：显示器没有该端口（如单 DP 口的显示器尝试切换到 DP-2）

**行为链**：
1. DDC 发送 `set 0x60 = 0x10`（DP-2）
2. 显示器固件处理方式因型号而异：
   - **方式 A**：忽略指令，DDC 回读仍为原端口 → 后端立即检测到不匹配
   - **方式 B**：短暂接受指令再回退 → 同场景 2 的行为

**软件检测**：
- 方式 A：后端首次验证即检测到不匹配 → `status: "warning"`（"切换失败：{target} 无信号，已自动恢复"）→ 前端回退按钮 → ✅ 可靠
- 方式 B：同场景 2，依赖 DDC 寄存器更新延迟 → ⚠️ 不可靠

**状态**：⚠️ 部分可靠（取决于固件行为）

---

### 场景 4：DDC 发送切换命令后通信中断

**条件**：切换后 DDC 无法回读（返回 None）

**典型例子**：显示器切换到其他输入后，DDC 通道暂时中断。

**行为链**：
1. DDC 发送切换命令 → 成功
2. DDC 回读 → 超时/失败（None）
3. 后端尝试回滚到原端口
4. 报告错误

**软件检测**：
- 后端首次验证返回 None → 触发回滚机制 → 报告错误
- ✅ 可靠检测

---

### 场景 5：DDC 发送切换命令本身失败

**条件**：m1ddc/ddc-hi 执行失败（权限问题、通信错误等）

**行为链**：
1. DDC 命令执行失败 → 返回非零退出码
2. 后端直接报告错误

**软件检测**：
- ✅ 完全可靠

---

### 场景 6：切换到当前已激活的端口

**条件**：用户点击当前正在使用的端口

**行为链**：
- 前端 UI 禁用当前端口按钮（disabled 状态）
- 不发送 DDC 命令

**软件检测**：
- ✅ 前端拦截，无需 DDC

---

## 三、场景检测能力总结

| 场景 | 描述 | DDC 可检测 | 当前软件支持 | 用户感知 |
|------|------|-----------|-------------|---------|
| 1 | 目标有信号 | ✅ | ✅ 可靠 | 切换成功 |
| 2 | 目标无信号（自动回退） | ⚠️ 延迟 | ⚠️ 先成功后静默修正 | 可接受 |
| 3A | 端口不存在（固件拒绝） | ✅ | ✅ 可靠 | 直接警告 |
| 3B | 端口不存在（固件先接受后回退） | ⚠️ 延迟 | ⚠️ 同场景2 | 可接受 |
| 4 | DDC 通信中断 | ✅ | ✅ 可靠 | 错误提示 |
| 5 | DDC 命令失败 | ✅ | ✅ 可靠 | 错误提示 |
| 6 | 切换到当前端口 | N/A | ✅ 前端拦截 | 按钮禁用 |

## 四、场景 2 的解决方案评估

场景 2 是唯一无法在合理时间内可靠检测的场景。以下是可选方案：

### 方案 A：延长验证等待时间

将后端+前端总验证时间延长到 15-20 秒。

| 优点 | 缺点 |
|------|------|
| 实现简单 | 所有切换操作都要等 15-20 秒，UX 极差 |
| 可能覆盖大部分显示器 | 仍无法保证覆盖所有固件延迟 |

**评价**：❌ 不推荐，对正常使用场景影响太大。

### 方案 B：不立即宣称成功，使用中性消息 + 持续监控

切换后不显示"已切换到 XXX"（暗示成功），而是：
1. 显示"切换指令已发送，正在监控状态..."（中性消息 + spinner）
2. 持续监控 15-30 秒
3. 监控期间检测到回退 → "XXX 无信号，已恢复到 YYY"
4. 监控期间未检测到回退 → "已确认切换到 XXX"

| 优点 | 缺点 |
|------|------|
| 不会出现"先成功后警告"的矛盾体验 | 正常切换也需要等待较长时间才确认 |
| 信息诚实准确 | UX 不够爽快 |

**评价**：⭐⭐⭐ 推荐。逻辑正确，信息不误导。

### 方案 C：乐观成功 + 轮询静默修正

切换后立即显示"已切换到 XXX"。如果后续轮询检测到回退，**静默更新 UI 按钮状态**，不再弹出额外的 toast 警告。

| 优点 | 缺点 |
|------|------|
| 切换体验最爽快 | 对无信号端口会短暂误导 |
| 正常使用场景零等待 | UI 状态会"自己变"，可能让用户困惑 |

**评价**：⭐⭐ 可接受。切换体验好，但信息不够准确。

### 方案 D：macOS 原生显示器事件辅助检测

利用 macOS CoreGraphics 的 `CGDisplayRegisterReconfigurationCallback`：
- 切换后如果显示器自动回退，Mac 可能检测到短暂的显示断开/重连事件
- 将此事件作为"可能发生了自动回退"的信号，延长验证等待或直接判定为回退

| 优点 | 缺点 |
|------|------|
| 不增加正常切换延迟 | 仅 macOS 可用 |
| 利用系统级信号而非纯 DDC | 不是所有显示器/回退都会触发此回调 |
| 可与方案 B 组合使用 | 实现复杂度中等 |

**评价**：⭐⭐⭐ 作为辅助手段有价值，但不能作为唯一依赖。

### 最初推荐方案：B + D 组合

1. **基础流程（方案 B）**：切换后显示中性的"正在确认"状态，持续监控 15 秒
2. **macOS 加速（方案 D）**：如果在监控期间收到 CoreGraphics 显示变化事件，立即判定为回退，提前终止等待
3. **正常切换优化**：如果切换到的是非 Mac 连接的端口（显示器从 Mac 侧"消失"），可提前确认成功

> **注**：方案 B+D 在实际测试中发现 15 秒等待时间过长，严重影响正常切换体验。经多轮迭代后采用方案 C，详见下一节。

## 五、当前实现方案（方案 C 改进版：即时乐观更新 + 轮询静默修正）

经多轮迭代和实际用户测试，最终采用**方案 C 改进版**，核心特点：按钮状态**点击后立即更新**，后端异步验证，失败时自动回退。

### 完整流程

```
用户点击切换按钮
  │
  ├─ [前端 · 即时] 立即更新按钮状态：
  │   ├─ 目标按钮变为"当前"（绿点 + 填充色）
  │   ├─ 原按钮恢复为普通状态
  │   ├─ 所有非目标按钮变为 disabled
  │   └─ 保存点击前的完整状态（用于回退）
  │
  ├─ [前端 · 即时] 显示 "正在切换输入源..." Toast（spinner）
  ├─ [前端] switchLock 阻止重复操作
  ├─ [前端 → 后端] 调用 cmd_switch_input（异步等待）
  │
  │  后端（全平台统一流程）：
  │  ├─ 获取 DDC_LOCK（串行化所有 DDC 操作）
  │  ├─ 读取当前输入（previous_input）
  │  ├─ 发送 DDC set 命令
  │  ├─ 第 1 轮验证：等待 600ms → DDC 读回
  │  │   ├─ 匹配目标 → 继续第 2 轮
  │  │   ├─ 不匹配 → 返回 warning（"切换失败：{target} 无信号，已自动恢复"）
  │  │   └─ None → 尝试回滚到 previous_input，返回 error
  │  ├─ 第 2 轮验证：等待 1400ms → DDC 读回
  │  │   ├─ 匹配目标 → 返回 success（"已切换到 {target}"）
  │  │   ├─ 不匹配 → 返回 warning
  │  │   └─ None → 标记未确认（短暂不可达）
  │  └─ 循环结束：
  │      ├─ 最后一轮已确认 → 返回 success
  │      └─ 最后一轮未确认（None）→ 返回 warning（"已发送命令但无法确认"）
  │
  ├─ [前端] 收到后端结果：
  │   ├─ success：
  │   │   ├─ 按钮保持不变（已是正确状态）
  │   │   ├─ Toast 变为"已切换到 XXX"（2.5s）
  │   │   └─ 清空 monitorsJsonRef，允许后续轮询修正
  │   │
  │   ├─ warning：
  │   │   ├─ 若后端返回 actual_input → 据此更新按钮状态（与硬件实际端口一致）
  │   │   ├─ 若无 actual_input → 回退到点击前的快照（monitorsSnapshotRef）
  │   │   ├─ Toast 显示警告信息（4s）
  │   │   ├─ 设置 monitorsJsonRef 为恢复后的状态 JSON
  │   │   └─ 启动 8 秒轮询冷却期（防止 DDC 过渡值覆盖正确的回退状态）
  │   │
  │   └─ error：
  │       ├─ 回退按钮状态
  │       ├─ Toast 显示错误信息（4s）
  │       └─ 启动 8 秒轮询冷却期
  │
  ├─ [前端] 释放 switchLock，恢复按钮可点击
  │
  └─ [前端] 后续轮询（每 3s）：
      ├─ 若在冷却期内 → 跳过本次轮询
      └─ 若冷却期已过 → 静默刷新监视器状态
          ├─ DDC 回读值与 UI 不一致 → 静默更新按钮状态（不弹 Toast）
          └─ 一致 → 无操作
```

### 关键设计决策

1. **按钮即时更新（真乐观更新）**
   - 点击后按钮**立即**变为"当前"状态，不等待后端 2-3 秒验证
   - 后端失败时自动回退到点击前的状态
   - 用户感知：正常切换（有信号端口）瞬间完成

2. **后端验证时间约 2s（全平台统一：两轮 600ms + 1400ms）**
   - 足以捕获大部分显示器固件立即拒绝的场景（端口不存在等）
   - 不足以捕获所有 DDC 寄存器延迟更新（某些显示器 >10s）

3. **Warning 结果携带 actual_input**
   - 后端 SwitchResult 包含 `actual_input: Option<u8>`，反映显示器实际端口
   - 前端优先使用 actual_input 更新 UI，避免使用快照中的过时信息
   - 若 actual_input 为 None（DDC 通信中断），才回退到快照
   - 消息格式：`"切换失败：{target} 无信号，已自动恢复到 {actual}"`

4. **8 秒轮询冷却期**
   - Warning/Error 回退后，前端设置 8 秒冷却期
   - 冷却期内 silentRefresh 不更新状态，防止 DDC 寄存器过渡值覆盖正确的回退状态
   - 冷却期过后恢复正常轮询

5. **前端不做扩展验证**
   - 早期版本有 7.5s 前端扩展验证循环，已移除
   - 原因：即使等待 10s+，仍有显示器的 DDC 寄存器未更新，无法完全消除误判

6. **轮询静默修正**
   - 若切换到无信号端口且后端未检测到（DDC 寄存器延迟更新）
   - 前端 3s 轮询最终检测到变化后，**静默更新按钮状态**，不弹额外 Toast
   - 用户看到的效果：按钮短暂显示目标端口，几秒后自动恢复为实际端口

7. **显示器自动回退的目标端口由固件决定**
   - 固件通常回退到**第一个检测到活跃信号的端口**
   - 例：从 HDMI-1 切换到无信号的 HDMI-2，显示器可能回退到 DP-1（Mac 连接的端口）而非 HDMI-1
   - 应用不发送额外 DDC 回滚命令（避免屏幕"闪两次"）
   - 应用只负责在轮询中检测到最终状态后静默更新 UI

### 方案权衡

| 优点 | 局限 |
|------|------|
| 正常切换（有信号端口）按钮瞬间更新 | 切换到无信号端口时，按钮先变过去再回退 |
| 不阻塞用户操作 | 后端验证期间（~2s）其他按钮不可点 |
| 避免了冗长等待和矛盾的消息序列 | 依赖轮询（冷却期后）来纠正漏检状态 |
| 不发送冗余 DDC 命令，避免屏幕闪烁 | 回退目标由固件决定，应用无法控制 |
| Warning 回退后 UI 立即稳定（8s 冷却） | 冷却期内无法检测到意外的端口变化 |
